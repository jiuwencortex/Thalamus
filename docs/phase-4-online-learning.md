# Phase 4 — Online Learning

> **Goal:** Replace synthetic matrix scores with real signal from actual interactions,
> and train a selection model that improves from every conversation.

---

## The limitation of synthetic scores

All the scores produced in Phases 0–2 are **synthetic**: generated by asking an LLM
to evaluate itself on LLM-generated queries. The process is:

```
LLM writes a query → LLM answers the query → LLM score compared to expected
```

This is useful for bootstrap — it gets the system running before real data exists.
But it has two structural weaknesses:

**1. The evaluator is the same as the evaluated.**
The same LLM that generates the expected answer also produces the candidate output.
It cannot surprise itself. Scores tend to cluster in a narrow range and miss real
failure modes.

**2. Generated queries may not match real user behavior.**
An LLM generating test queries for the `email-sender` skill will generate reasonable
email-related queries. But real users might ask in unexpected ways, combine it with
other tasks, or use it for purposes the skill description didn't anticipate.

Phase 4 collects real interaction data and uses it to:
1. Update the matrix scores with real-world evidence
2. Train a selection model that can predict good configurations from actual usage patterns

---

## What data to collect

For every agent conversation turn, log:

```json
{
  "turn_id": "uuid",
  "timestamp": "2025-10-14T09:12:00Z",
  "query_embedding": [...],       // vector, not the raw text (privacy)
  "context_config": {
    "skills": ["devops-toolkit", "code-reviewer"],
    "memory_sections": ["## Deployment setup"],
    "tools": ["bash_exec", "web_search"]
  },
  "outcome": {
    "explicit_rating": null,      // user thumbs up/down if given
    "implicit_signals": {
      "follow_up_correction": false,  // user sent "that's wrong, try again"
      "task_completed": true,         // agent reported task done, no complaint
      "conversation_length": 3        // how many turns to resolve the query
    },
    "component_usage": {
      "skills_used": ["devops-toolkit"],   // which skills were actually invoked
      "tools_called": ["bash_exec"]        // which tools were actually called
    }
  }
}
```

**Privacy note:** Store query embeddings (vectors), not raw query text. The embedding
captures the query's meaning without storing the user's exact words. This is the same
approach used in telemetry for other ML systems.

**Storage:** A simple JSONL file per agent instance, rotated weekly. No new database
needed for the initial version.

## Outcome quality signal

The outcome quality for a given turn is computed from the collected signals:

```python
def compute_outcome_quality(turn: dict) -> float:
    score = 0.5  # neutral baseline

    # Explicit rating (strongest signal)
    if turn["outcome"]["explicit_rating"] == "positive":
        score = 1.0
    elif turn["outcome"]["explicit_rating"] == "negative":
        score = 0.0

    # Implicit signals (weaker, combine additively)
    else:
        if turn["outcome"]["implicit_signals"]["task_completed"]:
            score += 0.2
        if turn["outcome"]["implicit_signals"]["follow_up_correction"]:
            score -= 0.3
        # Shorter conversations = faster resolution = better
        length = turn["outcome"]["implicit_signals"]["conversation_length"]
        score += max(0, 0.1 - 0.02 * length)

    return max(0.0, min(1.0, score))
```

This is intentionally simple. The exact formula can be tuned later.

## How real data improves the matrices

After collecting N turns (e.g. N=500), run a **matrix update**:

For each component C and each turn T where C was included in the context:

```
real_score(C, T) = outcome_quality(T)

new_synthetic_weight = max(0, 1 - n_real / n_needed)
new_real_weight = 1 - new_synthetic_weight

updated_score(C, query_type) = (
    new_synthetic_weight × synthetic_score(C, query_type)
  + new_real_weight × mean(real_score(C, T) for T with same query type as T)
)
```

As real data accumulates, it gradually replaces the synthetic scores.
When `n_real >= n_needed` (e.g. 100 real examples per component), the matrix entry
is fully based on real data.

This is a **soft replacement**: synthetic scores are used when real data is sparse,
real scores take over as data grows. The transition is smooth, not abrupt.

**No synthetic data is discarded.** It remains as the prior. Real data provides evidence
that updates the prior. This is the same idea as Bayesian updating.

## Training a selection policy

Beyond updating individual scores, Phase 4 trains a lightweight **selection policy**:
a small model that maps query type → which components to include.

### Why this is better than matrix lookup

Matrix lookup selects components independently. Component A scores 0.8, component B
scores 0.7 — both included. But in practice, A and B are always used together and the
combination gives a much better outcome than either alone. Or they are redundant and
only one is needed.

The selection policy learns this from data. It sees thousands of (query, config, outcome)
tuples and learns patterns like:
- "Queries about CI/CD always benefit from both devops-toolkit AND ## Deployment setup"
- "Web search and web fetch together are not much better than either alone for most queries"

### Model architecture

A simple linear model is sufficient for the first version:

```
Input:  query_embedding (e.g. 384 dimensions from a sentence-transformer model)
Output: probability vector of length N_components (one per skill/memory/tool)
Model:  Linear(384 → N_components) + Sigmoid activation
Loss:   Binary cross entropy between predicted probabilities and actual outcomes
```

Each output is the probability of including that component. At inference time,
threshold at 0.5 (or use the Pareto budget to pick a threshold).

This model:
- Is fast to train (no GPU needed for linear model)
- Is interpretable (each weight shows the relationship between query features and component inclusion)
- Generalizes from real data without overfitting (linear models are naturally regularized)

### Training

```
1. Collect turns until N_real >= 500 (configurable)
2. Build training set:
     X = [query_embedding for each turn]
     Y = [outcome_quality × inclusion_vector for each turn]
         # 1 where component was included AND outcome was good
         # 0 where component was included but outcome was poor
         # 0 where component was not included (no signal either way)
3. Train: minimize binary cross entropy with L2 regularization
4. Save model weights to policy.pkl
5. At query time: load weights, compute dot product, threshold
```

Training runs offline (batch), triggered weekly or when N_new_turns >= 200.

### Where the policy integrates

The policy sits in front of the existing matrix-based selector:

```
User query
    │
    ▼
Embed query
    │
    ├── [Phase 3] Find nearest cluster → load pre-computed config
    │
    └── [Phase 4] Query selection policy → predicted inclusion probabilities
    │
    ▼
Combine: use policy output where confidence is high,
         fall back to Phase 3 cluster config where confidence is low
    │
    ▼
Assemble context → send to LLM
```

The policy does not replace Phase 3 — it refines it. Phase 3 gives the base config,
Phase 4 adjusts it based on learned patterns.

## Gradient descent and why it applies here

Phases 1–3 do not use gradient descent because their optimization targets
(matrix scores, evolutionary fitness) are not differentiable.

Phase 4 introduces a differentiable target: the selection policy model parameters `W`.

```
Loss = -Σ [outcome_quality(T) × log P(config(T) | query(T); W)]
```

where `P(config | query; W) = sigmoid(W × query_embedding)`.

This loss IS differentiable with respect to W. Standard gradient descent
(or Adam optimizer) applies directly. This is where gradient descent enters the system.

The connection to the broader optimization landscape:
- Phase 3 (evolutionary) handles the combinatorial part (which set of components)
- Phase 4 (gradient) handles the prediction part (given a query, predict the right set)

Both are needed. Evolutionary search finds good configurations; gradient descent trains
a fast predictor for those configurations.

## Cold start handling

When Phase 4 first runs, there is no real data. The system falls back gracefully:

| Real data available | Behavior |
|---|---|
| None | Use Phase 3 cluster configs (synthetic only) |
| N < 100 | Use Phase 3, blend in real data with low weight |
| N = 100–500 | Mix synthetic and real, policy not yet trained |
| N > 500 | Policy trained, partially replaces cluster configs |
| N > 2000 | Policy is the primary selector, cluster configs as fallback |

The thresholds are configurable. The system degrades gracefully at every stage.

## Implementation order

1. `turn_logger.py` — hook into agent events, log in JSONL format
2. `outcome_scorer.py` — compute quality from implicit/explicit signals
3. Run for 2–4 weeks, collect data
4. `matrix_updater.py` — blend real scores into existing matrices
5. `policy_model.py` + `policy_trainer.py` — train linear policy
6. `policy_selector.py` — integrate at query time
7. A/B comparison: policy selection vs Phase 3 cluster lookup

**Total new code: ~350 lines.** No new infrastructure — logs to file, trains on CPU.

## What Phase 4 unlocks

- Matrix scores reflect real usage, not just synthetic evaluation
- Selection improves automatically as the agent is used more
- The system becomes self-improving: good configurations are reinforced, bad ones fade
- The policy model is interpretable: inspecting its weights reveals which query features
  predict which components — useful for understanding the agent's behavior patterns
- Foundation for future RL: the outcome quality signal can become the reward signal
  in a reinforcement learning setup, enabling more sophisticated optimization
